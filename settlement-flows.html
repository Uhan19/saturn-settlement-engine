1 · A redemption request's life (this already exists today)

When a user wants to exit sUSDat, they don't get paid instantly. Their request becomes an NFT that moves through four states. The middle two transitions are what the engine automates.

user acts Requested

User calls requestRedeem(shares, minValue). Shares are held in the queue; an NFT is minted as the receipt. minValue = the least USDat the user will accept.

Saturn locks In Progress

Saturn picks a batch of requests and locks them. From here the user can only lower their minValue, so the batch stays stable while it settles.

Saturn pays Processed

processRequests runs: USDat arrives in the queue, each request gets its share, the escrowed sUSDat is burned. The money now waits for the user.

user acts Claimed

User calls claim: the USDat is transferred to their wallet and the NFT receipt is burned. Done.

2 · The key idea: one payment, three places the money can come from

Every settlement ends in the exact same on-chain call. The only real decision is which pot of USDat pays for it.

Method 1 · use Saturn's own USDat

Saturn buys the exposure

Saturn pays redeemers from its own wallet (the "backbook") and ends up owning the STRC they exited. Only done when Saturn actively wants more STRC — a business decision, kept as a human-set config flag. That flag is the top-level check in the engine's decision order.

Saturn's wallet ➜ queue ➜ redeemers
Saturn's STRC holdings go up by the same amount
Method 2 · use the vault's own USDat

Pay from the vault's idle cash

The vault usually holds idle USDat — mostly recent deposits not yet converted to STRC. That USDat pays the redeemers directly. Nothing is bought or sold in the real world, so it's instant and works on weekends. The default rung when the buy-STRC flag is off.

vault's idle USDat ➜ queue ➜ redeemers
on paper, the vault "buys" the leavers' STRC from them at the oracle price
Method 3 · sell STRC for real

Go to the market

The last resort, when there isn't enough cash in the treasury or the vault: sell STRC through the broker, wait for the cash (T+1), bring it back on-chain as USDC, mint USDat, then pay. Takes days and crosses systems Saturn doesn't control.

broker sale ➜ cash ➜ USDC ➜ mint USDat ➜ queue ➜ redeemers

All three funnel into the same call: processRequests(batch, totalUsdat, strcSold, price) — which independently re-checks prices and every user's minValue before paying anyone.

4 · How the engine chooses the money source

Not an optimizer — a fixed order of preference with limits set by humans. The "do we want STRC" flag is the top-level check; the market is the last resort.

5 · The engine's heartbeat: the same five moves, every 30–60 seconds

The engine never trusts its own memory. Each cycle it rebuilds its picture of the world from scratch and only then acts.

1 · Read the chain

Queue requests and states, vault balances, oracle price.

2 · Read Postgres

What did I plan? What did I already start? Any half-done work?

3 · Read the broker

Order status, fills, cash — wherever an API exists.

4 · Compare

What is vs. what should be. Anything out of line becomes a to-do or an alarm.

5 · Act once, safely

At most one next action per settlement, written down before and after.

⟲ …then start over from step 1
Why this shape: if the engine crashes at any moment — even mid-settlement — nothing is lost. The next cycle re-reads everything and continues from whatever is actually true. Recovery isn't a special case; it's the only code path. And the queue contract itself refuses to pay the same request twice, so even a duplicated transaction fails harmlessly.

6 · Method 3 up close: the multi-day workflow

Each settlement that needs a real STRC sale walks this sequence of states, stored in Postgres. Every step forward is confirmed either automatically (chain or broker API) or by the operator entering a reference number. Every state has a maximum expected duration — sit too long and someone gets paged. It can stall, but never silently.

PLANNEDengine

batch chosen, amounts computed

LOCKEDon-chain

requests frozen; mins can only go down

SELL_ORDEREDbroker API

order placed with a unique ID so a retry can't sell twice

SOLDbroker API

fill confirmed; dollar amount now fixed

PROCEEDS_SETTLEDbroker API

T+1: cash actually lands

OFF_RAMPEDhuman

wire sent; operator enters the reference #

USDAT_MINTEDon-chain

USDC → USDat via M0

FUNDEDon-chain

USDat sits in the processor wallet, ready

PROCESSEDon-chain

processRequests pays the batch

RECONCILEDengine

closes only when broker, bank and chain numbers all agree

The known risk in this flow: the sale fixes a dollar amount early, but the on-chain settlement is price-checked against the oracle later. If STRC moves a lot in between, the batch can land outside the allowed tolerance and get stuck. The fix is speed — every day cut from the cash path shrinks this window — plus re-checking the tolerance right before submitting, which the notes call out in the method 3 signals.

8 · What keeps it safe

Every limit exists twice

Spending caps live in the engine's risk checks and as signing rules on the Fireblocks key itself. If the engine has a bug — or is compromised — the key still refuses to sign beyond the limits. The engine's mistake cannot exceed the key's allowance.

Three independent kill switches

1Engine off-flag — one config change and it stops planning and sending.
2Revoke the Fireblocks rules — the key stops signing, whatever the engine wants.
3Pause the contract — the queue itself stops, on-chain.

Any one of the three works even if the other two are unavailable.

The rule behind everything: every failure should end with the engine doing nothing — never with the engine doing the wrong thing. In this system waiting is always safe. A stalled settlement costs time; a wrong one costs money and trust.
